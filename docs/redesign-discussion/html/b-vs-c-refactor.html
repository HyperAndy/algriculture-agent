中等重构 vs 彻底重构

当前建议采用 B 中等重构；C 作为未来平台化蓝图保留，不建议现在直接做。

目录骨架

  • `src/app` 只保留路由和页面组装。
  • `src/features/*` 承载业务模块、查询、mutation、ViewModel。
  • `src/domain/*` 承载风险规则、作物规则、Agent 契约。
  • `src/design-system/big-screen` 和 `src/design-system/console` 分离主题。

适用原因

  • 保留现有 MVP 和数据库，交付风险可控。
  • 能解决当前最大问题：样式混乱、组件过大、业务逻辑散落。
  • 后续可以把 Agent Runtime 或数据服务单独拆出去。

服务边界

  • Web 只负责页面、交互和前端状态。
  • API Service 负责权限、业务聚合、数据写入、报表查询。
  • Agent Worker 负责长链推理、异步分析、任务生成和重试。
  • 数据层迁移到 Postgres，并预留多租户、审计、权限模型。

不建议现在做的原因

  • 会引入队列、部署、鉴权、服务通信等额外复杂度。
  • 资金申请阶段更需要稳定可演示，而不是完整工程平台。
  • 现有功能还没沉淀清楚，过早服务化容易拆错边界。
维度B 中等重构C 彻底重构
周期短到中，适合当前继续迭代长，需要重新规划里程碑
风险可控，保留现有功能和数据高，容易影响演示和交付节奏
质量收益显著改善可维护性和视觉体系最高，但依赖完整工程投入
推荐定位当前执行方案二期或正式产品化蓝图
